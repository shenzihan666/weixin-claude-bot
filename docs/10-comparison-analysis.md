# 10 — 原项目 vs 当前项目：精简设计与教学价值

> 从 3756 行生产级框架插件到 550 行教学型独立应用的逆向工程与重构

## 概览

本项目的诞生源于对 `@tencent-weixin/openclaw-weixin` npm 包（MIT License）的深度源码分析。我们通过逆向工程解析了腾讯 iLink 协议的完整实现，并将其重构成一个**极度精简、教学导向**的独立应用。

| 维度 | **原项目 (`@tencent-weixin/openclaw-weixin`)** | **当前项目 (`weixin-claude-bot`)** |
|------|----------------------------------------------|-----------------------------------|
| **项目性质** | OpenClaw 框架的微信 Channel 插件 | 独立 Node.js 应用 |
| **代码规模** | 33 个源文件，约 3756 行 TypeScript | 7 个源文件，约 550 行 TypeScript |
| **设计目标** | 生产环境完整功能 | 教学演示 + 核心功能最小集 |
| **架构耦合** | 深度依赖 OpenClaw 框架 SDK | 零框架依赖，纯 Node.js + TypeScript |
| **协议覆盖** | iLink 协议 100% 功能 | iLink 协议核心 20% 功能（文本消息） |
| **学习曲线** | 高（需理解 OpenClaw 架构） | 低（直接阅读核心逻辑） |

## 代码精简策略

### 1. 直接砍掉不需要的功能

| 模块 | 原项目行数 | 当前项目 | 决策理由 |
|------|-----------|---------|---------|
| CDN 媒体加密上传/下载 | ~400行 | 0行 | 纯文本 Bot 不需要文件传输 |
| 媒体处理管线 | ~300行 | 0行 | 只处理文本消息，不需要图片/语音 |
| 多账号管理 | ~350行 | 0行 | 单用户场景足够教学 |
| SILK 语音转码 | ~150行 | 0行 | 不需要语音识别转写 |
| 调试模式与日志 | ~200行 | 0行 | 教学项目不需要生产监控 |

**精简原则**：如果功能不是**绝对必要**，就直接删除。我们的核心目标是**微信消息 ↔ Claude Code 桥接**，不是完整的微信客户端。

### 2. 框架耦合代码重写

原项目深度依赖 OpenClaw 框架的 `ChannelPlugin` 接口：

```typescript
// 原项目：框架耦合
import { ChannelPlugin, PluginContext } from '@openclaw/core';

export class WeixinChannel implements ChannelPlugin {
  async initialize(ctx: PluginContext): Promise<void> { /* ... */ }
  async start(): Promise<void> { /* ... */ }
  async stop(): Promise<void> { /* ... */ }
}
```

我们将其重写为独立的 Node.js 应用：

```typescript
// 当前项目：独立应用
async function main() {
  const creds = loadCredentials();
  const config = loadConfig();

  while (true) {
    const resp = await getUpdates(api, { get_updates_buf: syncBuf });
    // 处理消息...
  }
}
```

**重写收益**：
- **零依赖**：不需要安装 OpenClaw 框架
- **透明性**：所有逻辑都在一个文件中可见
- **可调试性**：直接运行，没有中间层

### 3. 保留核心协议逻辑

虽然砍掉了 80% 的代码，但我们**保留了 100% 的核心协议知识**：

| 关键协议细节 | 原项目实现 | 当前项目实现 | 重要性 |
|-------------|-----------|------------|--------|
| `from_user_id` 必须为空字符串 | ✅ | ✅ | 填错会导致后续消息全部失败 |
| `client_id` 必须唯一 | ✅ | ✅ | 缺失会导致服务端去重丢弃消息 |
| `context_token` 每条消息都不同 | ✅ | ✅ | 必须缓存最新的，否则回复失败 |
| AES key 有两种编码格式 | ✅ | ❌（无媒体功能） | 媒体解密需要 |
| `errcode: -14` = session 过期 | ✅ | ✅ | 必须处理，否则无限失败循环 |
| Long-poll AbortError 不是错误 | ✅ | ✅ | 正确处理，避免重连风暴 |

这些细节是**从 3756 行源码中逆向发现的未文档化行为**，没有官方文档记录。我们保留了所有这些关键 edge case 的处理。

## 从原项目中学到的关键教训

通过分析 3756 行源码，我们发现了**微信 iLink 协议的关键设计**：

### 1. 消息送达的三要素

```typescript
// 这三个字段决定了消息能否成功送达
const msg: WeixinMessage = {
  from_user_id: "",          // 必须为空字符串！
  client_id: generateClientId(), // 必须唯一，防止去重
  context_token: latestToken,    // 必须是最新的
};
```

**踩坑记录**：
- `from_user_id` 填了 bot ID → 第一条能发，后续全部失败
- `client_id` 不唯一 → 服务端去重导致消息丢失
- `context_token` 使用旧的 → 消息无法送达

### 2. Session 过期的优雅处理

```typescript
const SESSION_EXPIRED_ERRCODE = -14;
const SESSION_PAUSE_MS = 60 * 60 * 1000; // 1小时

if (resp.errcode === SESSION_EXPIRED_ERRCODE) {
  console.error("Session 过期，暂停 1 小时后重试...");
  await sleep(SESSION_PAUSE_MS);
  continue;
}
```

**设计智慧**：原项目实现了完整的 session guard 机制，我们简化为 1 小时冷却期。

### 3. CDN 媒体的加密方案

虽然我们砍掉了媒体功能，但理解了其**安全设计**：

- **端到端加密**：文件在客户端 AES-128-ECB 加密后才上传
- **密钥分离**：加密密钥与文件分开传输
- **双重编码**：AES key 有两种编码格式（base64 或 hex）
